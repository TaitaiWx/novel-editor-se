# 版本管理设计

## 背景

小说编辑器的版本系统必须满足 4 个前提：

- 用户无需预装 Git、LFS 或任何外部工具
- 文本、图片、音频、PDF 等素材统一纳入版本快照
- 本地优先、离线可用、单机开箱即用
- UI 面向作者，而不是面向开发者的仓库工作流

基于这些约束，版本管理已从 Git 方案迁移为 **SQLite 原生快照方案**。

## 最终方案

### 方案概述

项目目录内维护一个隐藏的 `.novel-editor/` 目录，内部存放：

- `novel-editor.db`：SQLite 数据库
- 业务表：作品、角色、场景、大纲、设定、统计
- 版本表：快照、快照文件清单、去重后的 Blob 内容

应用不再依赖 `.git`、`git-lfs` 或系统命令行工具。

### 为什么不用 Git

对于作者型产品，Git 的问题不是能力不够，而是产品成本过高：

- 需要用户理解仓库、提交、rebase 等概念
- 默认依赖系统环境，无法保证开箱即用
- 业务数据在 SQLite，文本素材在 Git，容易形成双数据源
- 删除版本、改名版本等需求在 Git 中本质上是“改写历史”，不适合普通用户

### 为什么使用 SQLite 原生快照

SQLite 更适合本产品的核心场景：

- 嵌入式数据库，无需安装服务或额外运行时
- WAL 模式支持高频写入与稳定恢复
- 事务保证快照写入原子性
- 可以把业务结构和版本元数据放在同一数据库内
- 对文本和二进制内容都能统一建模

## 架构设计

### 层次结构

```
┌─ UI 层（渲染进程）──────────────────────────┐
│  版本历史弹窗 │ DiffEditor │ 素材/文件浏览器 │
├─ IPC 层 ───────────────────────────────────┤
│  db-version-create │ db-version-list │ ... │
├─ 服务层（主进程 + @novel-editor/store）────┤
│  versionOps / novelOps / characterOps      │
├─ 持久化层 ─────────────────────────────────┤
│  SQLite (metadata + dedup blob storage)    │
└────────────────────────────────────────────┘
```

### 表结构

版本功能新增 3 张核心表：

1. `version_snapshots`
   保存一次版本保存的元信息：消息、文件数、总字节数、创建时间。

2. `version_entries`
   保存某个快照下有哪些文件，以及它们引用了哪个内容哈希。

3. `version_blobs`
   保存真正的内容体。文本和二进制内容都按 `sha256` 去重后存进 SQLite BLOB，避免重复写入相同素材。

### 大文件与二进制素材处理

当前采用的是 **内容寻址 + 去重 Blob** 方案：

- 每个文件在创建快照前先计算 `sha256`
- 若内容哈希已存在，则仅新增索引，不重复存内容
- `version_entries` 只保存文件路径到 Blob 的映射
- 删除快照后自动清理孤儿 Blob

这套方案对以下资源统一有效：

- 纯文本章节
- Markdown / JSON / 配置文件
- PNG / JPG / WebP / SVG
- MP3 / WAV / OGG
- PDF / ZIP / 其他二进制资料

目前版本预览分为四类：

- 文本资源：进入 DiffEditor 做并排差异对比
- 图片资源：展示“历史快照 vs 当前文件”的左右对比视图
- PDF 资源：展示多页缩略图、主预览区和页码跳转
- 音频资源：展示“历史快照 vs 当前文件”的双播放器和元信息面板
- 其他二进制资源：展示 MIME、大小等元信息对比

## IPC 设计

当前版本管理使用以下 IPC 通道：

```typescript
const VERSION_CHANNELS = [
  'db-version-create',
  'db-version-start-create',
  'db-version-job-status',
  'db-version-list',
  'db-version-delete',
  'db-version-rename',
  'db-version-get-file-content',
  'db-version-restore-file',
];
```

## 工作流

### 打开项目

1. 用户打开任意文件夹
2. 应用自动创建或打开 `<project>/.novel-editor/novel-editor.db`
3. 若该文件夹还没有项目记录，则自动注册到 `novels` 表

### 保存版本

1. 扫描项目文件，忽略 `.novel-editor`、`.git`、`node_modules`、构建产物等目录
2. 先收集文件描述与总字节数，再分批读取并计算哈希与 MIME 信息
3. 与最近一次快照比对，完全一致则不创建新快照
4. 有变化时写入 `version_snapshots`、`version_entries` 和必要的 `version_blobs`

大项目下，扫描和持久化都以异步任务执行，前端通过轮询任务状态显示进度条与已处理字节数。

### 查看历史

1. 版本弹窗按当前文件过滤快照列表
2. 选择历史版本后读取对应文件内容
3. 文本文件进入 DiffEditor
4. 图片进入左右对比视图，PDF 进入多页预览器，音频进入双播放器，其余二进制资源进入元信息对比面板
5. 用户可直接把当前文件恢复到所选快照版本

## 当前边界

已支持：

- 项目自动初始化 SQLite 版本库
- 手动创建版本
- 版本列表查看
- 重命名版本
- 删除版本
- 文本文件版本对比
- 图片左右版本对比
- PDF 多页缩略图与页码跳转
- 音频双播放器与元信息预览
- 通用二进制资源元信息对比面板
- 当前文件恢复到指定快照
- 后台异步快照任务与进度条
- 二进制资源快照入库与去重存储
- `sample-data/media/` 下的成对样例素材，便于手动验证预览与快照差异

暂未支持：

- 按文件夹范围或整项目范围恢复历史版本
- 远程同步

## 后续演进

1. 为音频预览增加波形、声道和采样率等更细粒度信息。
2. 引入增量快照缓存，减少大项目首次扫描后的重复 IO。
3. 为版本快照增加标签、星标和恢复点。
4. 把结构化业务数据（角色、大纲、设定）的变更纳入同一版本时间轴。
